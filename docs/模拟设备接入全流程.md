# 模拟设备接入全流程

本文档用于完成“模拟设备接入、数据采集、数据上传、规则判断、告警生成、告警处置”的完整验证。当前流程使用外部 PowerShell 脚本模拟一台设备主动采集并上传遥测数据，设备侧通过 `/api/telemetry/report` 上报数据，后端继续复用现有遥测、规则、告警和日志链路。

## 1. 验证目标

本流程验证以下能力：

1. 模拟设备能够从平台外部发起数据上传。
2. 后端能够接收温度、湿度、压力等遥测数据。
3. 平台能够保存历史遥测并更新设备在线状态。
4. 规则引擎能够对上传数据执行阈值判断。
5. 超过阈值后能够自动生成规则审计和告警记录。
6. 告警能够在页面或接口中完成处置，并写入操作日志。

## 3. 平台数据准备

进入“设备管理 / 设备管理”，确认至少存在一台设备。系统初始化数据中通常包含：

```text
注塑机温度传感器01
空压站边缘网关01
```

如果没有设备，可新增：

```text
设备名称：模拟温湿度设备01
设备编码：SIM-TH-001
设备 Key：sim-th-001
设备位置：实验室
设备状态：OFFLINE
```

进入“规则引擎 / 规则设计”，确认存在并启用如下规则：

```text
规则名称：温度超过80℃自动告警
指标：temperature
运算符：>
阈值：80
告警等级：HIGH
启用状态：启用
```

该规则用于验证高温数据上传后能否触发告警。

## 4. 执行模拟设备脚本

脚本位置：

```text
industrial-iot-platform/scripts/simulated-device-flow.ps1
```

在 PowerShell 中执行：

```powershell
cd D:\作业\工业互联网平台开发实践\industrial-iot-platform
powershell -ExecutionPolicy Bypass -File .\scripts\simulated-device-flow.ps1
```

脚本默认会执行以下动作：

1. 登录平台并获取 Token。
2. 自动读取第一台设备作为接入设备。
3. 读取设备 `deviceKey`，模拟设备身份。
4. 通过 `/api/telemetry/report` 构造并上传一组正常遥测数据。
5. 通过 `/api/telemetry/report` 构造并上传一组高温遥测数据。
6. 查询历史遥测、规则审计和 OPEN 告警。
7. 在终端输出关键验证结果。

如需指定设备 ID：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\simulated-device-flow.ps1 -DeviceId 1
```

如需指定设备 Key：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\simulated-device-flow.ps1 -DeviceKey "key-th-001"
```

如需自动处置最新 OPEN 告警：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\simulated-device-flow.ps1 -DeviceId 1 -CloseAlarm
```

如后端网关端口不是 `8080`：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\simulated-device-flow.ps1 -BaseUrl "http://localhost:8080"
```

## 5. 手工接口验证

如果不使用脚本，也可以手工调用接口完成流程。

登录：

```powershell
curl -Method POST http://localhost:8080/api/auth/login `
  -ContentType "application/json" `
  -Body '{"username":"admin","password":"123456"}'
```

设备上报接口不要求后台用户 Token。上传正常数据：

```powershell
curl -Method POST http://localhost:8080/api/telemetry/report `
  -ContentType "application/json" `
  -Body '{"deviceKey":"key-th-001","temperature":36.5,"humidity":55,"pressure":101.3}'
```

上传高温数据：

```powershell
curl -Method POST http://localhost:8080/api/telemetry/report `
  -ContentType "application/json" `
  -Body '{"deviceKey":"key-th-001","temperature":86.2,"humidity":48,"pressure":101.1}'
```

查询历史遥测：

```powershell
curl -Method GET http://localhost:8080/api/telemetry `
  -Headers @{Authorization="Bearer panda-iot-demo-token"}
```

查询规则审计：

```powershell
curl -Method GET http://localhost:8080/api/rule-audits `
  -Headers @{Authorization="Bearer panda-iot-demo-token"}
```

查询告警：

```powershell
curl -Method GET http://localhost:8080/api/alarms `
  -Headers @{Authorization="Bearer panda-iot-demo-token"}
```

处置告警：

```powershell
curl -Method POST http://localhost:8080/api/alarms/1/handle `
  -Headers @{Authorization="Bearer panda-iot-demo-token"} `
  -ContentType "application/json" `
  -Body '{"handler":"admin","remark":"模拟设备全流程测试完成，告警已确认处置"}'
```

其中 `/api/alarms/1/handle` 中的 `1` 需要替换为实际 OPEN 告警 ID。

## 6. 页面检查点

执行脚本或手工接口调用后，前端页面应出现以下变化：

| 页面 | 检查内容 | 预期结果 |
| --- | --- | --- |
| 设备管理 / 设备管理 | 查看接入设备状态 | 设备状态变为 ONLINE，最后上线时间更新 |
| 数据中心 / 历史数据 | 查看遥测数据 | 能看到 temperature、humidity、pressure 记录 |
| 规则引擎 / 规则审计 | 查看规则执行结果 | 正常数据为 PASS，高温数据为 HIT |
| 数据中心 / 历史告警 | 查看告警列表 | 高温数据生成 HIGH 等级 OPEN 告警 |
| 数据中心 / 历史告警 | 处置告警 | 告警状态变为 CLOSED |
| 系统设置 / 操作日志 | 查看审计记录 | 能看到模拟上报和告警处置记录 |
| 控制台 | 查看统计变化 | 今日消息数、告警数和趋势图发生变化 |

## 7. 流程说明

模拟设备脚本本质上充当外部设备程序。它不通过前端页面点击“模拟上报”，而是在平台外部直接调用 `/api/telemetry/report` 上传数据。这样可以更清晰地体现设备侧主动接入平台的过程。

完整链路如下：

```text
模拟设备脚本
-> 采集正常遥测数据
-> 调用 /api/telemetry/report 上传
-> 后端保存历史遥测
-> 更新设备在线状态
-> 执行启用规则
-> 写入 PASS 规则审计
-> 采集高温遥测数据
-> 再次上传
-> 写入 HIT 规则审计
-> 自动生成 HIGH 告警
-> 告警处置
-> 写入操作日志
-> 控制台统计同步变化
```

## 8. 汇报说明

项目汇报时可以这样描述：

```text
系统通过外部模拟设备程序完成设备接入和数据采集验证。模拟设备采集温度、湿度、压力等遥测数据后，通过 HTTP 接口主动上传到平台。后端接收数据后完成设备在线状态更新、遥测数据持久化、规则阈值判断、规则审计写入和告警自动生成。运维人员可以在历史告警页面完成告警处置，处置结果同步写入操作日志。通过该流程，平台实现了从设备接入、数据采集、数据上传到告警处置的完整闭环。
```
